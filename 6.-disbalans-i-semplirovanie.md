Представьте, что вы обучаете модель предсказывать мошеннические транзакции. Из миллиона операций мошеннических — всего 1000, то есть 0.1%. Что произойдёт, если модель просто скажет «нет мошенничества» на каждую транзакцию? Её точность (accuracy) будет 99.9%. Звучит отлично. Но мошенники никак не пострадают.

Это и есть проблема дисбаланса классов.

{% hint style="info" %}
**Дисбаланс классов (Class Imbalance)** — ситуация, когда один класс встречается значительно реже другого.

Типичные примеры в бизнесе:
* Покупка после email-рассылки: 2–5% — да, 95–98% — нет
* Мошеннические транзакции: < 1%
* Отток клиентов: 5–15%
* Клик по баннеру: 0.1–2%

Дисбаланс — это **свойство задачи**, а не ошибка данных.
{% endhint %}

### 6.1 Почему accuracy — плохая метрика при дисбалансе

В ситуации 95% / 5% модель, которая всегда предсказывает большинство, получит 95% точности — и при этом будет абсолютно бесполезной.

```python
import numpy as np
from sklearn.metrics import accuracy_score, classification_report

y_true = np.array([0] * 950 + [1] * 50)
y_pred_dummy = np.zeros(1000)

accuracy = accuracy_score(y_true, y_pred_dummy)
print(f"Accuracy: {accuracy:.2%}")

print(classification_report(y_true, y_pred_dummy, target_names=['нет', 'да']))
```

**Правильные метрики при дисбалансе:**

| Метрика | Что измеряет |
| ------- | ------------ |
| ROC-AUC | Качество ранжирования; не зависит от порога |
| PR-AUC | Точность и полнота; особенно хороша при сильном дисбалансе |
| Recall | Сколько объектов minority мы нашли |
| Precision | Из тех, что нашли — сколько были правильными |
| F1-score | Гармоническое среднее Recall и Precision |

### 6.2 Undersampling — уменьшение большинства

Идея: если у нас слишком много объектов большого класса, берём случайную часть из них.

```python
from imblearn.under_sampling import RandomUnderSampler
import numpy as np
from collections import Counter

X = np.random.randn(1050, 5)
y = np.array([0] * 1000 + [1] * 50)

print(f"До: {Counter(y)}")

rus = RandomUnderSampler(random_state=42)
X_res, y_res = rus.fit_resample(X, y)

print(f"После: {Counter(y_res)}")
```

**Плюсы:** быстро, уменьшает дисбаланс\
**Минусы:** теряем данные — выбрасываем большую часть обучающей выборки

### 6.3 Oversampling — увеличение меньшинства

Дублируем объекты редкого класса, чтобы их стало больше.

```python
from imblearn.over_sampling import RandomOverSampler

print(f"До: {Counter(y)}")

ros = RandomOverSampler(random_state=42)
X_res, y_res = ros.fit_resample(X, y)

print(f"После: {Counter(y_res)}")
```

**Плюсы:** не теряем данные большого класса\
**Минусы:** риск переобучения — модель видит одни и те же примеры много раз

### 6.4 SMOTE — синтетические примеры

SMOTE создаёт **новые синтетические примеры**, а не просто копирует существующие.

Как это работает:
1. Берём объект из minority-класса
2. Находим его K ближайших соседей из того же класса
3. Генерируем новую точку где-то между ними

```python
from imblearn.over_sampling import SMOTE

smote = SMOTE(random_state=42)
X_res, y_res = smote.fit_resample(X, y)

print(f"После SMOTE: {Counter(y_res)}")
```

{% hint style="info" %}
SMOTE создаёт новые точки между соседями, а не копирует существующие. Это снижает риск переобучения по сравнению с обычным Oversampling.

Ограничение: SMOTE плохо работает **с категориальными признаками**. Для смешанных данных используйте **SMOTENC**.
{% endhint %}

### 6.5 Взвешивание классов (Class Weights)

Самый элегантный подход: не меняем данные, а говорим модели, что **ошибки на редком классе стоят дороже**.

```python
from sklearn.linear_model import LogisticRegression
from sklearn.metrics import classification_report

model_balanced = LogisticRegression(class_weight='balanced')
model_balanced.fit(X_train, y_train)

model_manual = LogisticRegression(class_weight={0: 1, 1: 20})
model_manual.fit(X_train, y_train)

print(classification_report(y_test, model_balanced.predict(X_test)))
```

**Плюсы:** не меняем данные, стабильно, легко включается\
**Минусы:** вероятности становятся «сдвинутыми» — нужна калибровка

### 6.6 Изменение порога (Threshold Moving)

По умолчанию модели классифицируют объект как класс 1, если вероятность > 0.5. При дисбалансе этот порог можно и нужно сдвигать.

```python
import numpy as np
import pandas as pd
from sklearn.metrics import f1_score, precision_score, recall_score

y_proba = model_balanced.predict_proba(X_test)[:, 1]
thresholds = np.arange(0.05, 0.95, 0.05)
results = []

for threshold in thresholds:
    y_pred = (y_proba >= threshold).astype(int)
    results.append({
        'threshold': round(threshold, 2),
        'precision': precision_score(y_test, y_pred, zero_division=0),
        'recall':    recall_score(y_test, y_pred),
        'f1':        f1_score(y_test, y_pred, zero_division=0)
    })

results_df = pd.DataFrame(results)
best_threshold = results_df.loc[results_df['f1'].idxmax(), 'threshold']
print(f"Лучший порог по F1: {best_threshold}")
```

{% hint style="info" %}
Выбор порога — это **бизнес-решение**, а не техническое.

* Антифрод: лучше заблокировать 10 честных клиентов, чем пропустить 1 мошенника → снижаем порог, повышаем Recall
* Email-кампания: лучше отправить письмо только тем, кто точно купит → повышаем порог, повышаем Precision
{% endhint %}

### 6.7 Где применять сэмплирование

{% hint style="danger" %}
**Сэмплирование применяется только к обучающей выборке — никогда к validation или test.**

**Правильно — через imblearn Pipeline:**

```python
from imblearn.pipeline import Pipeline as ImbPipeline
from imblearn.over_sampling import SMOTE
from sklearn.linear_model import LogisticRegression
from sklearn.model_selection import cross_val_score

pipeline = ImbPipeline([
    ('smote', SMOTE(random_state=42)),
    ('model', LogisticRegression())
])

scores = cross_val_score(pipeline, X_train, y_train, cv=5, scoring='roc_auc')
print(f"ROC-AUC: {scores.mean():.4f}")
```

**Неправильно:**

```python
smote = SMOTE()
X_resampled, y_resampled = smote.fit_resample(X, y)
```
{% endhint %}

***

| Метод | Плюсы | Минусы | Когда использовать |
| ----- | ----- | ------ | ------------------ |
| Undersampling | Быстро, уменьшает дисбаланс | Теряем данные | Когда данных очень много |
| Oversampling | Не теряем majority данные | Риск переобучения | Когда данных мало |
| SMOTE | Синтетические примеры | Не работает с категориальными | Числовые данные, умеренный дисбаланс |
| Class Weights | Не меняем данные, стабильно | Вероятности смещены | Универсальный вариант |
| Threshold Moving | Гибко под бизнес-задачу | Нужна интерпретация | Когда важен баланс Precision/Recall |

{% hint style="info" %}
Дисбаланс — это **свойство задачи**, а не ошибка данных. Иногда достаточно правильно выбрать метрику и сдвинуть порог — и модель будет отлично работать без дополнительных манипуляций с данными.
{% endhint %}
